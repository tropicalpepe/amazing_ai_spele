# Deterministic Two-Player Game Assignment

## Overview and Objective
The objective of this assignment is to apply your knowledge of search-based problem solving by developing software for a deterministic, two-player game with perfect information (Computer vs. Human). Your team is free to choose any programming language and development environment for this project.

---

## The Specific Game Rules
Your team has been assigned the following game. 
*(Note: You are permitted to modify these rules if you encounter undefined states, if the end state becomes unreachable, or if the initial rules result in a game tree so simple that it can be fully generated, which would defeat the purpose of creating a heuristic evaluation function).*

### Game Setup
1. **Initial Numbers:** At the start of the game, the software must randomly generate 5 numbers in the range of **10,000 to 20,000**. These initial numbers must be perfectly divisible by both 2 and 3 (i.e., divisible by 6).
2. **Player Selection:** The human player chooses one of the 5 generated numbers to start the game.
3. **Points and Bank:** Both players start with **0 points**. There is also a "Game Bank" that starts at **0 points**.

### Gameplay
* Players take turns making moves. 
* In each turn, the current player must divide the current number by either **2** or **3**.
* **Restriction:** A division is only allowed if the resulting number is a whole number (integer).
* **Scoring:**
    * If the number is divided by **2**, the **opponent** receives **2 points**.
    * If the number is divided by **3**, the **current player** receives **3 points**.
* **Bank Addition:** If the result of the division is a number that ends in **0 or 5**, **1 point** is added to the Game Bank.

### Endgame and Winning
* The game ends as soon as the current number is **less than or equal to 10**.
* The player whose move triggers the end of the game (the one who makes the final division) collects the Bank, adding all Bank points to their own score.
* **Victory:** The player with the most points wins. If both players have the same amount of points, the game is a tie.

## Software & GUI Requirements
Command-line interface (CLI) applications will **not** be accepted. Your software must include a Graphical User Interface (GUI). It does not need to be a complex 3D interface; standard visual elements like menus, buttons, text fields, icons, and lists are perfectly fine.

The user interface must allow the human player to:
* Choose who makes the first move (Human or Computer).
* Choose which algorithm the computer will use for that specific game: **Minimax** or **Alpha-Beta Pruning**.
* Execute their moves and visually see the updates on the game board (or screen) after both human and computer turns.
* Restart and play a new game once the current game is finished.
* *Note: There is no requirement to visually display the actual game tree in the GUI.*

---

## Technical & Algorithmic Requirements
Your team must design, implement, and test the following technical components:

1. **Data Structure:** You must store the generated part of the game tree using appropriate data structures (e.g., classes, linked lists, objects). Pure variable collections are not allowed.
2. **Tree Generation:** The game tree must be generated up to a specified depth limit (lookahead over *n* moves), depending on the game's complexity and your available computing resources.
3. **Heuristic Evaluation Function:** You must design and implement a custom heuristic function to evaluate non-terminal board states when the computer needs to make a move.
4. **Algorithms:** Both the **Minimax** and **Alpha-Beta Pruning** algorithms must be fully implemented to search the tree and decide the computer's moves.

---

## Required Experiments
Once the software is built, your team must conduct **10 experiments for each algorithm** (20 games total) and record the following metrics:
* Number of wins for the Computer vs. the Human.
* Total number of tree nodes generated by the computer.
* Total number of tree nodes evaluated by the computer.
* The computer's average time taken to execute a move.

## Automatic Rejection Criteria
Your project will not be reviewed, and you will not be allowed to defend it, if any of the following occur:
* The team applies for the defense but fails to submit the project report.
* The submitted software is a command-line application.
* The project lacks a proper data structure for storing the game tree.
* The game tree is pre-defined, hardcoded, or loaded from a file rather than generated dynamically by the code.
